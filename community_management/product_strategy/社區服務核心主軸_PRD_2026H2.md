# 社區服務核心主軸 PRD（2026 H2）

**一句話核心摘要：以「金流催繳效率（B）+ 高齡代辦履約（A）」作為 12 週 MVP 雙核心，並內建「資安信任層（F）+ 租客協作流程（J-MVP）」降低導入阻力，目標在首波試點社區驗證可複製的營收與續用模型。**

**版本：** v0.1  
**撰寫人：** PM（草稿）  
**最後更新：** 2026-06-09  
**狀態：** 草稿  
**文件來源主軸：** [社區服務產品方向選手池（2026 H2）](./社區服務產品方向選手池_2026H2.md)

## 一、為什麼做？（Problem Statement）

### 背景
台灣社區服務市場已出現「需求長期上升、平台寡占、供給分散」三個同時存在的特徵。若直接複製大而全社區 App，將在龍頭優勢場景中消耗資源；相反地，先切入可量化 ROI 的治理/金流場景與可感知價值的高齡服務場景，更有機會建立替代方案。

依既有策略結論，本案 12 個月主軸為：  
1) 以 B（金流催繳）建立決策者可見的經濟價值；  
2) 以 A（高齡代辦）建立住戶可感知的差異化；  
3) 以 F（資安信任層）避免信任風險；  
4) 以 J（租客體驗）補齊角色覆蓋缺口。

### 用戶痛點
- 管委會/物業財務：催繳效率低、對帳流程長、欠費風險預警不足。
- 第一線管理員/保全：工單與服務回報分散，異常責任不清，住戶抱怨集中在櫃台。
- 高齡住戶與家屬：需要可托付且可追蹤的代辦流程，現有流程透明度低。
- 租客/房東/物業三方：入退租、報修責任與權限管理經常落在人工作業，成本高且容易爭議。

### 商業目標（12 週 MVP）
- 在 5-10 個試點社區落地 MVP，驗證「可導入、可使用、可續用」。
- 以「收繳率改善 + 服務效率提升」作為導入與續約主要銷售語言。
- 建立至少 2 個可對外複製的試點案例，作為 Next 階段擴張素材。

| 物業公司管理層 | 擴點與續約關鍵角色，關注標準化與可複製導入 | P0 |
| 高齡住戶 | 服務主要受益者，關注易用與可靠性 | P1 |
| 外地子女/照護家屬 | 高齡服務付費潛力角色，關注可視化與通知 | P1 |

### 代表性用戶故事（User Story）
> As a 管委會財委,  
> I want to 即時看到收繳狀態與催繳進度，並可自動分層通知欠費戶，  
> So that 我能在 8 週內提升收繳率並降低人工催收工時。

> As a 高齡住戶家屬,  
> I want to 看到代辦服務每個節點的履約狀態與完成證據，  
> So that 我可以放心委託社區服務且願意持續使用。

## 三、做什麼？（Feature Scope）

### 3.1 產品定義（MVP）
MVP 為「社區履約與現金流雙核心平台」，包含四個模組：
1. **B-MVP：金流催繳效率模組（必做）**
2. **A-MVP：高齡代辦履約模組（必做）**
3. **F-Foundation：資安信任基線（必做）**
4. **J-Lite：租客入退租與報修責任分流（必做但輕量）**

### 3.2 範圍內（In-Scope）

| # | 功能描述 | 優先級 | 備註 |
|---|----------|--------|------|
| 1 | 分層催繳規則（提醒節奏/訊息模板/欠費分群） | P0 | B-MVP 核心 |
| 2 | 收繳儀表板（收繳率、逾期戶、催收進度） | P0 | 決策者視圖 |
| 3 | 對帳狀態追蹤（已繳/待核/異常） | P0 | 不含複雜財會整併 |
| 4 | 高齡代辦建立與派工（維修/清潔/採買） | P0 | A-MVP 核心流程 |
| 5 | 服務履約證據（時間戳、照片/備註、結案） | P0 | 建立可追責與信任 |
| 6 | 家屬通知（受理/進行中/完成） | P1 | A 差異化關鍵 |
| 7 | 角色權限分級（管委會/物業/住戶/家屬） | P0 | F-Foundation |
| 8 | 敏感資料遮罩與操作留痕 | P0 | F-Foundation |
| 9 | 租客入退租權限開關（J-Lite） | P1 | 試點流程驗證 |
| 10 | 報修責任分流（租客/房東/公設） | P1 | J-Lite 基本流程 |

### 3.3 範圍外（Out-of-Scope）
以下功能本次不做：
- 競品資料自動搬家工具（G 完整版）— 本版先以手動/半自動導入為主。
- 零廣告純訂閱品牌方案（H）— 需待付費意願驗證後再啟動。
- 大型硬體整合（門禁 IoT、監控設備深度串接）— 本版聚焦軟體流程價值。
- 全量租賃生命週期產品（J 完整版）— 本版僅驗證入退租與報修責任分流。

## 四、怎麼做？（Functional Requirements）

### 4.1 主要流程

#### 流程 A：金流催繳（B-MVP）
1. 管委會設定催繳規則（逾期天數、通知節奏、通知對象）。
2. 系統每日產生欠費分群名單與通知任務。
3. 住戶完成繳費後，狀態自動更新至儀表板。
4. 財委可檢視收繳率趨勢與異常案件，並導出週報。

#### 流程 B：高齡代辦履約（A-MVP）
1. 住戶/家屬建立代辦需求（類型、時間、特殊備註）。
2. 管理員或合作供應商接單並更新進度。
3. 完成後上傳履約證據並結案。
4. 家屬收到完成通知，可回饋滿意度。

#### 流程 C：租客協作（J-Lite）
1. 管理員建立租客入租資料，系統自動開通對應權限。
2. 退租時一鍵關閉租客權限並保留操作紀錄。
3. 租客報修時選擇責任類型，系統自動分流到房東/物業/公設流程。

### 4.2 功能細節

#### [B1] 分層催繳規則
- **觸發條件：** 每日排程檢查欠費狀態，或財委手動重跑。
- **系統行為：** 依規則推送通知並更新催收階段。
- **邊界情況：**
  - 若住戶已繳款但狀態未同步，標記為「待核」並提示人工覆核。
  - 若通知連續失敗，轉為管理員人工聯繫任務。

#### [A1] 代辦工單與履約證據
- **觸發條件：** 住戶/家屬提交代辦需求。
- **系統行為：** 產生工單、派工、進度通知、結案存證。
- **邊界情況：**
  - 超過 SLA 未更新進度時，系統升級提醒管理員。
  - 若工單取消，需保留取消原因與責任記錄。

#### [F1] 權限與稽核
- **觸發條件：** 使用者查看或修改敏感資料。
- **系統行為：** 依角色授權，記錄操作日誌，敏感欄位遮罩顯示。
- **邊界情況：**
  - 無權限操作需回傳明確拒絕訊息並記錄事件。
  - 管理員批次匯出資料需有審計記錄。

#### [J1] 入退租權限開關與責任分流
- **觸發條件：** 建立租約/退租/租客報修。
- **系統行為：** 自動開關權限、套用責任分流規則。
- **邊界情況：**
  - 權限異常（重複帳號/過期租約）需進入人工覆核佇列。
  - 三方責任爭議案件可手動改派，保留改派軌跡。

## 五、非功能需求（Non-Functional Requirements）

| 項目 | 要求 |
|------|------|
| 效能 | 核心頁面載入 < 2 秒（P95） |
| 可用性 | 月可用性目標 99.9% |
| 安全性 | 權限分級、敏感資料遮罩、操作稽核、異常事件告警 |
| 可觀測性 | 需具備催繳流程、工單流程、通知送達率的事件追蹤 |
| 多語系 | 本版以繁中為主，預留多語欄位 |
| 行動裝置 | 行動優先（住戶端）、桌面優先（管理端） |

## 六、成功指標（Success Metrics）

### 6.1 北極星與核心指標（MVP）
**北極星指標（MVP 階段）：** 試點社區「月度可驗證履約完成數」（含催繳閉環 + 代辦結案）

| 指標 | 現況（Baseline） | 8-12 週目標 | 測量方式 |
|------|-----------------|------------|----------|
| 管理費收繳率 | 試點啟動前各社區基線 | +8% 以上 | 社區月報 + 系統儀表板 |
| 催繳人工工時 | 試點前 4 週平均 | -20% 以上 | 管理員工時回報 |
| 服務結案時效 | 試點前平均結案時間 | -30% 以上 | 工單時間戳 |
| 包裹/服務相關客訴率 | 試點前月平均 | -25% 以上 | 客服工單分類 |
| 租客入退租客服工單 | 試點前月平均 | -30% 以上 | J-Lite 事件追蹤 |
| 試點社區續用意願 | N/A | 70% 以上 | 試點結束決策訪談 |

### 6.2 護欄指標（Guardrail Metrics）
- 系統通知送達率不可低於 98%。
- 權限誤開放事件需維持 0 起重大事故。
- 高齡代辦取消率若連續 2 週高於 15%，需啟動流程修正。

## 八、假設與風險（Assumptions & Risks）

### 假設
- 試點社區願意提供基線資料（收繳率、客服工單、結案時間）供前後對比。
- 物業與管委會可接受 MVP 初期流程調整，並配合操作教育。
- 家屬願意訂閱/使用高齡代辦通知功能（需於試點驗證）。

### 風險

| 風險描述 | 可能性 | 影響程度 | 緩解方案 |
|----------|--------|----------|----------|
| 第一線操作抗拒（流程變複雜） | 中 | 高 | 優先簡化管理端 UX；每週蒐集第一線回饋並快迭代 |
| 資料品質不足導致 KPI 無法比較 | 中 | 高 | 試點前建立資料盤點清單與標準化欄位 |
| 高齡服務供給品質不穩 | 中 | 高 | 建立 SLA 與供應商評分機制；先從少量服務類型起跑 |
| 權限設定錯誤造成信任風險 | 低 | 高 | 預設最小權限 + 重要操作雙重確認 + 稽核告警 |
| J-Lite 規則不清造成三方爭議 | 中 | 中 | 導入前定義責任矩陣與可追溯改派流程 |

## 九、對齊與待決策事項

### 9.1 需管理層確認
1. MVP 的試點社區選擇標準（戶數、租賃比例、既有工具狀態）。
2. B/A 的資源配比（建議 6:4，先確保金流 ROI 可快速出結果）。
3. J-Lite 是否納入首版必交項（建議納入，避免後續補洞成本）。

### 9.2 跨部門依賴
- **BD/Ops：** 試點招募、導入教育、每週營運回顧。
- **Engineering：** 金流狀態一致性、通知可靠性、權限稽核能力。
- **Design：** 管理端高頻流程一屏可完成，降低第一線負擔。
- **Legal/Compliance：** 個資處理、委外服務履約責任條款。
